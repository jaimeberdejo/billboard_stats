# Stack Research

**Project:** Billboard Stats — Next.js Frontend  
**Researched:** 2026-04-27  
**Mode:** Ecosystem — "What is the standard 2025/2026 stack for Next.js + Neon + Vercel?"

## Recommended Stack

### Core Framework

| Library | Version | Purpose | Rationale |
|---------|---------|---------|-----------|
| `next` | 15.3.x | Full-stack React framework | Current stable release. App Router is the default and recommended path. Turbopack is now the default dev bundler. React 19 support is included. |
| `react` / `react-dom` | 19.x | UI rendering | Ships with Next.js 15; async Server Components require React 19. |
| `typescript` | 5.x | Type safety | Built in to `create-next-app`. Required for async Server Components (needs TS 5.1.3+). Use `next.config.ts` (not `.js`) for typed config. |

### Database

| Library | Version | Purpose | Rationale |
|---------|---------|---------|-----------|
| `pg` (node-postgres) | 8.13.x | PostgreSQL client | **Primary recommendation for Vercel Fluid.** Neon's own docs explicitly recommend `pg` over `@neondatabase/serverless` for Vercel deployments because Fluid Compute allows TCP connection reuse, making pooled TCP the lowest-latency approach. |
| `@vercel/functions` | latest | Connection pool lifecycle | Provides `attachDatabasePool(pool)` — tells Fluid Compute to keep the `Pool` alive across concurrent requests and drain it cleanly on shutdown. Required companion to `pg` on Vercel Fluid. |
| `@types/pg` | 8.x | TypeScript types for pg | Dev-only. Provides types for `Pool`, `Client`, `QueryResult`. |

**Connection pattern (copy this into `src/lib/db.ts`):**

```ts
import { Pool } from 'pg';
import { attachDatabasePool } from '@vercel/functions';

const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  ssl: { rejectUnauthorized: false }, // required by Neon
});

attachDatabasePool(pool);

export default pool;
```

Each route handler pulls a client from the pool, executes queries, and releases it:

```ts
// app/api/charts/route.ts
import pool from '@/lib/db';
import { NextRequest, NextResponse } from 'next/server';

export async function GET(request: NextRequest) {
  const client = await pool.connect();
  try {
    const { rows } = await client.query('SELECT ...', []);
    return NextResponse.json(rows);
  } finally {
    client.release();
  }
}
```

### Styling

| Library | Version | Purpose | Rationale |
|---------|---------|---------|-----------|
| `tailwindcss` | 4.x | Utility-first CSS | Default in `create-next-app` since Next.js 15.3. v4 uses CSS-first config (`@theme` in `globals.css`); no `tailwind.config.js` generated by default. 3-10x faster full builds. |
| `@tailwindcss/postcss` | 4.x | PostCSS integration | Still required for Next.js (Vite gets zero-config; Next.js still needs a thin `postcss.config.mjs`). |


Then in `globals.css` with Tailwind v4:

```css

@theme {
  --font-sans: var(--font-space-grotesk), sans-serif;
  --color-brand: #C8102E;

```

### App Router Data Fetching Pattern

This is a read-only public app — no auth, no mutations. The recommended pattern is:

| Approach | When to use | This project |
|----------|-------------|-------------|
| Server Components querying DB directly | Pages that render server-side data | YES — for static/ISR pages like Records |
| Route Handlers (`app/api/**/route.ts`) | Endpoints that need to be called client-side (search, chart toggles) | YES — for interactive pages (search results, chart week selector) |
| Server Actions | Form submissions, mutations | NO — app is read-only |

**Critical:** In Next.js 15, `GET` Route Handlers are NOT cached by default (changed from v14). For read-only endpoints that can be cached (e.g., latest chart), add:

```ts
export const dynamic = 'force-static'; // or use revalidate
export const revalidate = 3600; // seconds
```

For endpoints that must be fresh on every request (e.g., search), no configuration needed — dynamic by default.

### Vercel Configuration

| Setting | Value | Why |
|---------|-------|-----|
| Environment variable | `DATABASE_URL` | Neon's modern integration sets this (pooled). Manual setup: copy from Neon dashboard → Project → Connect. |
| `DATABASE_URL_UNPOOLED` | Direct TCP, no pgBouncer | Needed if you run migrations directly (not needed in v1 — ETL handles schema). |
| Node.js runtime | 20.x (Vercel default) | Neon serverless driver 1.0+ requires Node 19+; `pg` 8.x works on 18+. Either way, Vercel defaults to 20. |
| `vercel.json` | Not needed initially | Only add if you need custom headers (CSP), redirects, or region pinning. |

**`.env.local` for local development:**

```
DATABASE_URL="postgresql://<user>:<password>@<endpoint>.neon.tech/<dbname>?sslmode=require&channel_binding=require"
```

The `sslmode=require&channel_binding=require` parameters are required by Neon (not optional).

## Key Decisions

### 1. `pg` + `@vercel/functions` over `@neondatabase/serverless`

**Decision:** Use `pg` with `attachDatabasePool`.

**Why:** Neon's own documentation explicitly recommends `pg` + TCP pooling for Vercel Fluid Compute, not `@neondatabase/serverless`. The Neon serverless driver routes queries over HTTP/WebSocket — optimal when TCP is unavailable (Cloudflare Workers, classic serverless with no warm instances). Vercel Fluid keeps functions warm and allows connection reuse, making direct TCP "the lowest-latency and most performant" method once a connection is established. The HTTP-based approach has constant per-query overhead; pooled TCP eliminates it on warm invocations.

**When you'd reverse this:** If you explicitly opt into Vercel Edge Runtime for any route (`export const runtime = 'edge'`), TCP is unavailable and you must use `@neondatabase/serverless`. For this app, standard Node.js runtime is fine for all routes.

### 2. Tailwind CSS v4 over v3

**Decision:** Use Tailwind v4 (ships by default with Next.js 15.3).

**Why:** `create-next-app` now scaffolds v4. CSS-first config is simpler for a design-token-heavy app (Billboard red `#C8102E`, custom font). v3's JS config adds ceremony. v4's `@theme` block in `globals.css` is the right place to define brand tokens.

**Caveat:** v4 has real breaking changes from v3. Tailwind provides a `@tailwindcss/upgrade` codemod. Starting greenfield, this is not a concern.

### 3. Server Components for page rendering + Route Handlers for client-driven queries

**Decision:** Do not create a separate REST API layer for Server Components. Query `pool` directly in `page.tsx` for server-rendered data. Use Route Handlers only for endpoints fetched client-side (e.g., search-as-you-type, chart week selection without navigation).

**Why:** Adding an extra `fetch()` hop from a Server Component to your own API route adds latency for no benefit — the Server Component already runs on the server. Next.js core team explicitly discourages this pattern.

### 4. No ORM

**Decision:** Use raw `pg` queries, not Prisma or Drizzle.

**Why:** The existing Python service layer has dozens of hand-tuned SQL queries including `pg_trgm` fuzzy search, complex `JOIN`s, and aggregate stats. These would require significant ORM-specific adaptation with no benefit. The queries are already validated against the schema. TypeScript types can be defined manually or generated once from the schema if needed. An ORM would add a build step, migration overhead, and potential query-shaping friction.

## What NOT to Use

| Technology | Why Not |
|------------|---------|
| `@neondatabase/serverless` as primary driver | Correct choice is `pg` + Fluid pooling for Vercel. Keep this as a fallback only if you add Edge Runtime routes. |
| `@vercel/postgres` | Deprecated — Vercel migrated its Postgres product to Neon. The `@vercel/postgres` package still works but wraps Neon; use Neon directly to avoid the indirection. |
| Prisma or Drizzle ORM | Adds migration complexity and query reshaping overhead for a project with existing, validated SQL. The queries are the asset; don't abstract them away. |
| Pages Router | Superseded by App Router since Next.js 13. All new Next.js 15 projects use App Router. |
| `getServerSideProps` / `getStaticProps` | Pages Router patterns — irrelevant in App Router. Equivalent is `async` Server Components + `dynamic`/`revalidate` config. |
| CSS Modules or styled-components | Tailwind v4 covers everything needed. The HTML prototype is already utility-class-friendly. CSS Modules add file overhead; styled-components requires client-side JS. |
| `next/headers` in Route Handlers for DB config | Do not read env vars inside handlers per-request. Initialize `Pool` once at module scope in `lib/db.ts` — this is what makes Fluid reuse work. |
