<task type="checkpoint:human-action">
  <name>Task 2: Dump local Billboard DB and restore into Neon</name>
  <read_first>
    - .planning/phases/05-i-want-to-configure-neon-and-deploy-the-app-in-vercel/05-RESEARCH.md (sections: Data Migration, Common Pitfalls 1 and 2)
  </read_first>
  <what-built>
    This is a human-action checkpoint. The data migration requires running pg_dump and pg_restore with your local PostgreSQL credentials. The executor cannot access your local PostgreSQL instance or the Neon credentials.
  </what-built>
  <how-to-verify>
    Run these commands in your terminal from the project root:

    **Step 1 — Add pg tools to PATH for this session:**
    ```
    export PATH="/opt/homebrew/opt/postgresql@16/bin:$PATH"
    which pg_dump
    ```
    Expected: `/opt/homebrew/opt/postgresql@16/bin/pg_dump`

    **Step 2 — Dump local database (takes ~2-5 minutes for 193 MB):**
    ```
    pg_dump \
      -Fc \
      -v \
      -Z 1 \
      --lock-wait-timeout=20s \
      -d "postgresql://postgres@localhost/billboard" \
      -f billboard.dump
    ```
    If prompted for a password and you don't have one set, try: `-d "postgresql://postgres@localhost/billboard"` — psql on macOS often allows local connections without a password for the postgres user.

    Expected: verbose output showing each table being dumped, ending with no ERROR lines. A file `billboard.dump` appears in the project root.

    Verify dump file size:
    ```
    ls -lh billboard.dump
    ```
    Expected: file size ~30-60 MB (compressed from 193 MB raw).

    **Step 3 — Restore into Neon (uses UNPOOLED connection string — do NOT use the pooled one):**
    ```
    pg_restore \
      -v \
      -O \
      --no-acl \
      --single-transaction \
      -d "postgresql://USER:PASS@ep-<id>.us-east-1.aws.neon.tech/neondb" \
      billboard.dump
    ```
    Replace with your actual UNPOOLED Neon connection string from Task 1.
    This takes ~5-15 minutes for 193 MB over the network.

    Expected: verbose output showing CREATE TABLE, COPY, CREATE INDEX statements. Extension-ownership ERRORs (for pg_trgm or plpgsql) are non-fatal and can be ignored if they appear — the data still loads.

    A clean exit with no "ERROR" lines on data tables means the restore succeeded.
  </how-to-verify>
  <resume-signal>Type "done" when pg_restore completes without data errors, or describe any error.</resume-signal>
</task>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local shell → pg_dump | pg_dump reads credentials from connection URI in shell history |
| pg_restore → Neon | Connection string with credentials transmitted over TLS |
| billboard.dump (disk) | Intermediate artifact on local filesystem contains full dataset |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | pg_dump connection URI in shell history | mitigate | Shell history will contain the local connection string `postgresql://postgres@localhost/billboard` — this has no password, so risk is low. For Neon connection string in pg_restore, warn user to use a password manager or `.pgpass` file. Plan instructions note the risk. |
| T-05-02 | Information Disclosure | billboard.dump on local filesystem | mitigate | File contains full dataset (not PII — public Billboard chart data). Instruction in Task 3 to `rm billboard.dump` after successful restore eliminates the artifact. |
| T-05-03 | Spoofing | Neon connection string entered interactively | accept | User types/pastes the string themselves in their own terminal session. No third-party involvement. Risk is low for a solo developer workflow. |
| T-05-04 | Denial of Service | Neon free tier 0.5 GB limit | accept | 193 MB dataset is well within the 0.5 GB limit. No mitigation needed for v1.0. |
</threat_model>
