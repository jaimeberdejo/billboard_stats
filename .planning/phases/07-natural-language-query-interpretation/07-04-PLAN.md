<objective>
Close the three actionable UAT gaps from Phase 7 UAT.md (gaps 1, 2, and 4 — gap 3 was accepted as intentional behaviour).

Gap 1 (minor): package.json is missing "type": "module", causing Node.js to emit a
[MODULE_TYPELESS_PACKAGE_JSON] performance warning every time the fixture verifier runs.

Gap 2 (major): "songs by taylor swift" returns needs_clarification instead of search.
The ARTIST_FILTER_RE at interpret.ts:48 matches "by taylor swift", setting artistNames
non-null and therefore making recordsCuePresent=true even when no chart, metric,
weeksMin, peakRange, or debutRange is present. The fix is to exclude artistNames from
the recordsCuePresent gate unless at least one other hard records cue is also present.

Gap 4 (major): "songs from last week" returns needs_clarification instead of unsupported.
detectUnsupportedCue only matches 4-digit years via YEAR_FILTER_RE; relative date phrases
("last week", "this week", "last month", "today", "yesterday", "recent") are not caught,
so the query falls through to the records path. The fix is to add a RELATIVE_DATE_RE
pattern (or extend UNSUPPORTED_CUE_WORDS) to catch those phrases early.

Purpose: All three fixes are deterministic and low-risk. They tighten gate conditions
without touching the happy-path logic. Two new golden fixtures lock in the corrected
behaviours so they cannot regress.

Output: Updated package.json, interpret.ts, catalog.ts, fixtures.ts; passing fixture
verifier with no warnings; passing build and lint.
</objective>

STEP C — Fix Gap 2 (artist-only query sent to records path) in src/lib/nlq/interpret.ts

The bug is in buildCustomInterpretation at line 340. The current code:

```typescript
const recordsCuePresent =
  explicitChart !== null ||
  artistEntity !== null ||
  artistNames !== null ||          // <-- problem: artistNames alone triggers records
  weeksMin !== null ||
  peakRange.min !== null ||
  debutRange.min !== null ||
  /\b(entries|weeks|peak|debut|position|top\s+\d+)\b/.test(normalizedQuestion);
```

Replace lines 340-347 with the guarded version. artistNames must not count as a records
cue unless at least one genuine records signal is also present. Introduce a helper
boolean that captures all the hard cues, then fold artistNames in conditionally:

```typescript
const hardRecordsCue =
  explicitChart !== null ||
  artistEntity !== null ||
  weeksMin !== null ||
  peakRange.min !== null ||
  debutRange.min !== null ||
  /\b(entries|weeks|peak|debut|position|top\s+\d+)\b/.test(normalizedQuestion);

const recordsCuePresent = hardRecordsCue || (artistNames !== null && hardRecordsCue);
```

Wait — the second branch reduces to `hardRecordsCue` again because `artistNames !== null
&& hardRecordsCue` is only true when `hardRecordsCue` is already true. The correct
minimal expression is simply:

```typescript
const hardRecordsCue =
  explicitChart !== null ||
  artistEntity !== null ||
  weeksMin !== null ||
  peakRange.min !== null ||
  debutRange.min !== null ||
  /\b(entries|weeks|peak|debut|position|top\s+\d+)\b/.test(normalizedQuestion);

const recordsCuePresent = hardRecordsCue;
```

This intentionally removes `artistNames !== null` from the gate entirely. artistNames is
still extracted and used in the final RecordsCustomInterpretation payload (line 551) when
a real records query is confirmed. The only change is that artistNames alone no longer
forces a query into the records path.
